## Context

当前项目运行在 `Vite + React + Tailwind`，应用通过 `App.tsx` 中的 `step` 状态在首页、编辑器、预览区、帮助页和管理入口之间切换。首页由 `LandingPage.tsx` 承担，当前实现延续了上一轮“像素工作台”方向：浅色背景、控制台式版面、主题卡片入口和偏工具化的信息分区。

这和本次 change 的目标存在明显偏差。用户给出的两张参考图都指向同一类首页体验：深色背景、像素世界观 hero、产品站导航、搜索与分类筛选、课程卡片网格，以及“游戏菜单式”的探索感。首页的核心任务应当从“进入工作台”调整为“发现内容和进入路径”，但应用内的编辑、预览、复制、草稿同步等核心工作流仍然要保持现状。

现有首页还承担了两个技术约束：

- 首页点击模板后可以直接进入编辑台，因此新首页不能切断当前 `templates` 数据与 `onSelectTheme` / `onEnterStudio` 行为。
- 当前项目已经存在一套较强的像素类与 `neo-*` token，若直接全局替换为新的 dark pixel 体系，容易影响编辑器和预览区，因此首页的视觉升级需要优先采用“首页范围内收敛”的方式。

## Goals / Non-Goals

**Goals:**

- 将首页重构为暗色、游戏化、课程发现导向的 pixel catalog 体验，而不是浅色工具控制台。
- 建立首页专用的 dark pixel 视觉基线，明确深色背景、双字体层级、hero 场景、搜索/筛选控件和课程卡片风格。
- 保留当前首页与应用主流程的连接能力，包括主题选择、进入编辑台、打开文档和管理员隐藏入口。
- 在现有技术栈中用可维护的方式落地，优先复用共享组件能力，但允许首页建立独立的结构类和布局模式。
- 保持移动端可用性，包括筛选项横向滚动、卡片栅格自适应和 hero 内容的响应式裁切。

**Non-Goals:**

- 不在本次 change 中重构编辑器、预览页、帮助页或管理工作台的整体视觉方向。
- 不把整个应用重新切换成新的 dark pixel token 体系；本次主要约束首页。
- 不引入新的路由系统、状态管理方案或外部 UI 框架。
- 不新增真实课程系统、后端搜索 API 或课程详情页。

## Decisions

### 1. 首页改为“发现页”，但继续复用当前模板数据作为课程卡数据源

当前应用没有独立的课程实体，首页最接近“内容列表”的现有数据就是 `templates`。本次首页的目标是重构发现体验，而不是引入新的业务模型，因此设计上继续使用 `templates` 作为卡片数据源，但在首页展示语义上把它包装为“课程/模块目录”。

这样做的好处是：

- 不会引入新的后端依赖或数据层变更。
- 仍能保留点击卡片后进入编辑台的现有行为。
- 可以通过前端映射增加分类标签、难度、专题分组和 hero featured item，而不改变现有主题对象结构。

备选方案：

- 新建一套首页专用课程数据模型：视觉更纯粹，但会增加重复数据与维护成本。
- 继续使用当前“主题模板”语义：会让首页仍显得像工具入口，不符合参考图。

结论：继续复用现有模板数据，但在首页层建立新的展示映射。

### 2. 采用“首页作用域 dark pixel shell”，而不是覆盖全局样式基线

现有应用的编辑器和预览页已经依赖一套较重的像素样式类。若直接把新的暗色 token 替换为全局默认，会让非首页区域受到波及，也会把这个 change 扩展成新一轮全站样式重构。

因此本次设计选择在首页容器内建立独立的 shell 规则，例如：

- 首页背景、导航、hero 和卡片容器使用暗色 token。
- 双字体规则在首页层明确为“像素标题 + 可读 sans 正文”。
- 首页允许使用新的卡片边框、筛选 pill、搜索框和 CTA 组合，而不强制改写编辑器/预览区。

备选方案：

- 全局切换 dark pixel token：视觉统一度更高，但风险过大，且超出本 change 范围。
- 完全内联首页样式：短期快，但会形成大量一次性 class 组合。

结论：首页范围内引入新的 dark pixel 设计层，尽量复用共享组件，同时避免污染全局工作流页面。

### 3. 字体采用“双轨策略”：像素显示字体只用于标题/CTA，正文使用高可读 sans

用户提供的 prompt 明确要求首页呈现“Arcade-Style Typography”，但正文必须保持现代可读性。参考图也都符合这一点：大标题与 logo 使用像素字，正文、导航和卡片描述保持更顺畅的阅读节奏。

因此本次首页不继续沿用“中文全局像素字体”的思路，而是采用：

- Display / Hero / Logo / 少量短 CTA：`Press Start 2P` 或等价像素显示字体
- 正文 / 导航 / 说明 / 卡片描述：`Inter` 或当前运行时可用 sans-serif

备选方案：

- 延续上一轮全局 `Fusion Pixel` 策略：会显著削弱暗色课程目录的可读性，也偏离本次 prompt。
- 全部使用现代 sans：易读，但会失去参考图最关键的像素身份。

结论：采用双轨字体体系，并把像素字体使用范围限制在短文本和强调层级。

### 4. Hero 使用“世界地图 / 像素景观”结构，而不是控制台式信息面板

两张参考图虽然版式不同，但 hero 都依赖大面积 pixel art 场景建立情绪：一张偏横向课程世界横幅，一张偏地图式世界探索。它们的共同点不是功能密度，而是“先建立世界观，再引出探索入口”。

因此首页 hero 的设计决策是：

- 使用大面积 pixel art 背景或景观横幅作为首屏视觉中心。
- 文案区域保留在左侧或中左区域，覆盖一个主标题、一个简短副标题和一个主 CTA。
- 赞助方/支持方横幅作为 hero 后的次级带状区块，而不是嵌入到主内容网格中。

备选方案：

- 沿用当前像素天空 + 控制面板组合：过于工具化。
- 使用普通渐变 hero：会失去参考图的像素世界感。

结论：hero 以大场景 pixel art 为中心，功能入口后移到搜索和卡片区。

### 5. 首页信息架构采用“导航 → hero → sponsor strip → featured section → search/filter → course grid”

参考图实际上给出了两个互补信息架构：一张更强调搜索和分类，一张更强调专题区和课程分组。为了同时吸收两者优点，首页结构采用组合式信息架构：

1. 顶部导航：品牌、主导航、辅助图标、主 CTA
2. Hero：大场景 + 核心文案
3. Sponsor / world strip：支持方或氛围带
4. Featured section：一个专题标题，例如 “The Legend of Python”
5. Search / filters：搜索框 + pills 筛选
6. Course grid：2-4 列响应式课程卡片

这样既能保留产品首页的探索感，也能兼容当前“选择一个模板并进入编辑台”的主流程。

备选方案：

- 只保留 hero + 卡片网格：结构太平，缺少专题层级。
- 只保留搜索 + 筛选：更像课程市场页，不像有世界观的产品首页。

结论：采用组合式首页架构，平衡世界观、导航和发现效率。

### 6. 课程卡片维持现代结构，但用像素插画与轻量像素装饰建立风格

参考图中的卡片并不是厚重的像素按钮卡，而是相对现代的 dark card：圆角、细边框、轻悬浮、上半部分是 pixel art 图像。真正传达像素风的是插画、局部字体、色彩和少量结构细节，而不是所有边角都做成硬像素块。

因此课程卡片的设计取向是：

- 结构上偏现代 dark card，易于阅读和响应式排布。
- 顶部必须有 pixel art 封面图。
- 标题和描述保持可读 sans。
- Tag、难度 pill、hover 反馈更接近参考图里的“教育产品卡片”，而不是控制台面板。

备选方案：

- 用强硬像素边框重做所有卡片：会再次靠近上一版的控制台风格。
- 做成普通 SaaS 卡片：会丢失像素个性。

结论：卡片采用“现代结构 + 像素内容”的混合策略。

## Risks / Trade-offs

- [首页仍复用模板数据，语义不是原生课程模型] → 通过首页视图映射补充标签、专题分组与难度信息，不修改底层主题结构。
- [新 dark pixel shell 与现有全局像素类并存] → 将新的视觉规则限制在首页容器和首页相关组件内，避免扩散到编辑器与预览页。
- [pixel art 资源质量直接决定首页质感] → 优先复用现有缩略图并补充更符合场景的封面资源；在实现阶段把图像约束写入任务。
- [首页探索风格增强后，进入编辑台的直接性可能下降] → 在 hero 和卡片操作中保留清晰 CTA，确保主流程仍然一跳可达。
- [双字体系统与当前全局字体规则可能冲突] → 在首页层显式指定标题和正文字体，不依赖全局默认推断。

## Migration Plan

1. 先定义首页信息架构与视觉基线，明确哪些区块保留、哪些区块删除。
2. 为首页建立 dark pixel token 与结构类，优先作用于 `LandingPage` 及其子组件。
3. 将当前 `templates` 数据映射为 featured item、筛选标签和课程卡片列表。
4. 重构首页导航、hero、专题区、搜索与筛选、课程网格。
5. 验证首页到编辑台、文档入口和管理员入口的现有跳转仍然可用。

回滚策略：

- 若新首页影响主流程，可先回退 `LandingPage` 与相关首页样式映射，不动其他页面。
- 若 dark pixel token 影响到非首页区域，则将新增样式进一步收敛到首页私有类。

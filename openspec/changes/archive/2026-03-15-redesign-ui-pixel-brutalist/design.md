## Context

当前项目是基于 Vite + React + Tailwind 的单页应用，而不是 Next.js 应用。现有界面已经带有一定的新粗野主义和像素感元素，但颜色、字体、背景、按钮、输入框和页面区块仍然存在多套视觉规则并存的问题，导致首页、编辑器、预览区和共享组件之间缺少统一的设计语言。

本次变更的目标不是调整业务流程，而是建立一个可复用、可维护的复古像素风 UI 基线，并将关键页面迁移到这套系统中。用户需求中提到的 `tailwind.config.js`、全局字体接入、基础样式类提取和主要页面替换，都属于跨多个模块的前端重构，因此需要单独的设计文档来约束实现边界和技术路径。

约束如下：

- 当前运行时和构建工具为 Vite，不能直接按 Next.js 的 `layout.tsx` 或 `next/font/google` 方案照搬实现。
- 现有组件广泛依赖 Tailwind 原子类和 `neo-*` 视觉 token，重构需要避免一次性破坏 Markdown 编辑、预览和主题切换工作流。
- 现有 Tailwind 配置和 `index.css` 中已经定义了一套 `neo` 色板与阴影变量，需要决定是原地替换还是并存过渡。

## Goals / Non-Goals

**Goals:**

- 建立一套统一的复古像素风设计系统，包括颜色、字体、边框、阴影、背景纹理、状态色和偶数像素网格规范。
- 将基础视觉规则沉淀为全局可复用样式能力，至少覆盖 `pixel-panel`、`pixel-btn`、`pixel-input` 三类核心 UI 模式。
- 在不改变现有业务流程的前提下，重构主要页面和核心组件，使首页、编辑器和预览区遵循一致的视觉基线。
- 为后续 specs 和 implementation 提供明确的分层方案，减少“边改边想”的样式漂移。

**Non-Goals:**

- 不在本次变更中把项目从 Vite 迁移到 Next.js。
- 不调整 Markdown 渲染、主题提取、微信草稿同步或后端 API 行为。
- 不尝试在本次设计中保留所有旧 `neo` 视觉细节；旧风格只在必要的兼容层中短暂存在。
- 不将所有历史 demo、主题素材和文档示例在本阶段一起清理归档。

## Decisions

### 1. 保持现有 Vite 架构，只做 UI 设计系统重构

本次 change 只处理视觉系统和组件样式，不引入 Next.js 迁移。用户给出的目标更接近“用一套指定设计系统重构 UI”，而不是“更换应用框架”。如果在同一 change 中同时做框架迁移和 UI 重构，会让 proposal 中的 capability 边界失焦，也会让 tasks 难以验证。

备选方案：

- 直接迁移到 Next.js 后再做 UI 重构：可以满足 `next/font/google` 诉求，但会把工作量扩展到路由、构建、部署和环境变量兼容，超出本 change 范围。
- 保持 Vite 并适配等价方案：能够在最小架构扰动下完成视觉替换，也更符合当前项目状态。

结论：采用“保持 Vite，按等价方式实现像素设计系统”。

### 2. 用新的 pixel token 覆盖全局视觉基线，而不是继续扩展 `neo` 命名

需求中的颜色、边框和阴影规则已经明确，继续沿用 `neo-*` 命名会让新旧视觉语义混杂，降低可读性。因此设计上应新增一组清晰的像素主题 token，例如页面背景、面板背景、像素黑、六种功能色和交互高亮色，并将其作为新的全局基线。

备选方案：

- 直接修改现有 `neo-*` token 对应的值：改动表面更少，但会让旧名称承载新语义，后续维护成本更高。
- 新增 `pixel-*` token，并逐步替换页面和组件引用：迁移更清晰，也便于在过渡期识别未完成的旧样式。

结论：采用新的 `pixel-*` 设计 token，并通过全局样式与 Tailwind 主题扩展统一输出。

### 3. 基础视觉规则在全局样式中抽象为可复用类

像素面板、像素按钮、像素输入框都依赖固定的边框和硬阴影组合，且交互状态包含复杂的 inset shadow 与按下位移。继续把这类规则散落在 JSX 的长串 className 中，会明显增加维护成本，也不利于后续页面统一替换。

备选方案：

- 每个组件单独内联 Tailwind 任意值：实现快，但容易在按钮、输入框、卡片上产生多个相近但不相同的版本。
- 在全局 CSS 中用 `@layer components` 提供 `.pixel-panel`、`.pixel-btn`、`.pixel-input` 等复用类，再由页面局部追加差异样式：既满足需求，也更利于统一。

结论：基础视觉模式统一抽象到全局样式层，组件只负责组合，不重复定义核心阴影与边框逻辑。

### 4. 字体接入采用“当前框架可用方案”，并严格映射中英文层级

用户已经明确了字体系统：中文全局文本统一使用 `Fusion Pixel`，英文醒目大标题使用 `Press Start 2P`，英文次级文本和按钮使用 `Silkscreen`，并要求全局关闭抗锯齿。由于当前不是 Next.js，设计上应使用等价方案，例如通过本地字体资源或当前运行时可用的字体加载方式接入这些字体，并在全局变量中明确中文文本、英文大标题、英文次级文本/按钮三类字体角色。

备选方案：

- 为满足字体方案而强行引入 Next.js：与本次 change 范围不符。
- 在当前项目中使用本地字体资源或当前运行时可用的加载方式，并通过 Tailwind/自定义变量统一应用：能满足视觉需求，也不要求更换框架。

结论：当前 change 采用 Vite 可用的字体接入方案；实现中必须保留 `Fusion Pixel` / `Press Start 2P` / `Silkscreen` 的字体职责分工。若未来迁移到 Next.js，可在不改 spec 目标的前提下替换字体加载实现。

### 5. 版式尺寸采用偶数像素网格约束

用户明确要求所有字号、margin、padding 使用偶数像素值，以对齐底层像素网格。这意味着本次重构不只是替换颜色和字体，还需要审视共享组件和页面布局中的尺寸体系，避免出现 `10px` 之外的奇数间距、`15px` 字号或不规则内边距。

备选方案：

- 仅在新增像素组件中使用偶数值，旧页面保留原有尺寸：实现成本更低，但会让新旧区块视觉密度不一致。
- 以偶数像素网格作为全站重构约束，重写核心页面和共享组件的字号与间距：改动更大，但符合本次设计系统目标。

结论：采用偶数像素网格作为统一约束，至少覆盖共享组件和主路径页面。
### 6. 页面重构采用“先基座、后页面”的分层迁移方式

设计系统重构如果直接逐页散改，容易出现基础类还没稳定、页面已经开始复制样式的问题。更稳妥的做法是先完成 token 和基础类，再依次替换共享组件、最后替换首页与工作台等主要页面。

备选方案：

- 逐页面直接重写：视觉上推进快，但容易出现同一按钮在多个页面实现不同。
- 先设计系统、后共享组件、再页面迁移：更符合本次 change 的结构化目标。

结论：采用分层迁移，确保全站使用同一套 panel/button/input 规则。

## Risks / Trade-offs

- [需求文案基于 Next.js，但项目当前是 Vite] → 在 specs 和 implementation 中明确“按当前框架提供等价实现”，避免把框架迁移误纳入本变更。
- [旧 `neo-*` 样式与新 `pixel-*` 样式短期并存] → 采用分阶段替换，并在任务中优先覆盖共享组件和主路径页面，减少混用窗口。
- [Fusion Pixel 在长篇中文正文中的可读性可能弱于常规 UI 字体] → 保留必要的中文回退字体链，并在实现阶段验证正文字号、行高与段间距是否仍然可读。
- [硬阴影、厚边框和点阵背景可能影响已有布局密度] → 页面重构时同步调整间距、容器尺寸和按钮高度，不仅替换颜色和阴影。
- [偶数像素网格要求会扩大尺寸调整范围] → 在任务中单独检查字号、margin 和 padding，优先覆盖共享组件与核心页面，避免无序返工。
- [一次性改动多个前端入口区域，可能引入视觉回归] → 按“token → 基础类 → 共享组件 → 页面”的顺序落地，并在每阶段做构建和手动验证。

## Migration Plan

1. 在 Tailwind 配置和全局样式中新增像素主题 token、字体变量、偶数像素尺寸基线与基础复用类。
2. 更新共享 UI 组件，使按钮、输入框、基础卡片容器优先使用像素类。
3. 重构首页和核心工作台区域，统一布局、背景、边框、按钮和面板样式。
4. 检查预览区、帮助文档入口和辅助面板，清理明显残留的旧 `neo` 风格实现。
5. 运行构建并做核心路径验收，确认编辑、预览、复制等工作流未受影响。

回滚策略：

- 如果视觉重构影响主流程，可先回退页面级替换，保留 token 和基础类改动；
- 如果基础类定义本身存在问题，则整体回退本 change 的样式层修改。

- 是否需要在 specs 中明确背景点阵纹理属于推荐实现而非强制要求，以降低不同页面上的视觉负担？

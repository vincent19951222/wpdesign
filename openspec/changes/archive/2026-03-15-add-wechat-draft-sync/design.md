## Context

当前项目是一个纯前端 Markdown 排版工具，现有主路径是“编辑 Markdown → 预览 → 复制到公众号后台”。仓库中已经存在一个 Express `server/`，当前仅用于代理 Moonshot API 与保存主题文件。前端没有单独的文章标题字段，标题隐含在 Markdown 的首个 `# H1` 中；草稿同步如果直接复用现有正文渲染，将导致公众号标题栏与正文首屏标题重复。另一方面，微信公众号草稿接口属于服务端能力，公众号凭据与 access token 不能暴露在前端。

该 change 的目标是在不破坏现有手动复制链路的前提下，增加一个真实的公众号草稿 API 同步能力。v1 只支持经典像素 API-safe 主题，不做全主题兼容；正文和标题由现有编辑/预览链路提供，封面上传与摘要输入不做前端能力扩展。

## Goals / Non-Goals

**Goals:**
- 在现有 Express 服务中增加微信公众号草稿代理能力
- 在前端预览页增加“一键同步草稿”入口与状态反馈
- 使用经典像素 API-safe 主题作为 v1 唯一支持的同步主题
- 从 Markdown 首个 H1 派生草稿标题，并从正文 HTML 中移除重复标题模块
- 保留现有“预览 + 复制”路径不变

**Non-Goals:**
- 不支持所有主题直接同步
- 不实现封面上传 UI、摘要 UI、作者署名 UI
- 不实现自动发布、群发或浏览器自动化粘贴
- 不把微信公众号凭据暴露给前端

## Decisions

### 1. 复用现有 Express 服务承载微信公众号代理
前端继续保持静态应用，服务端在 `server/index.js` 中新增微信公众号代理路由，而不是引入第二套后端服务。

原因：
- 当前仓库已有可运行的 Express 服务，可直接承载安全凭据与代理逻辑
- 草稿同步需要服务端保存 `appsecret` 并调用微信接口，纯前端不可行
- 新增后端最小化部署变化，适合 v1 试点

备选方案：
- 新建独立服务：隔离更强，但会增加部署与运维复杂度，v1 不值得
- 直接前端调用微信接口：无法安全保存凭据，不可接受

### 2. 前端只在预览页提供草稿同步入口
草稿同步入口与复制按钮并列放在 `Preview` 页面，而不是编辑页或首页。

原因：
- 预览页已经有完整渲染后的 `#wechat-output` DOM，可直接生成同步正文
- 现有用户心智就是“预览后发布/复制”，新增同步入口成本最低
- 避免为编辑页额外构建一套独立的 HTML 渲染流程

备选方案：
- 在编辑页直接同步：需要新增独立的渲染/序列化管线，决策和维护成本更高

### 3. v1 同步仅支持经典像素 API-safe 主题，并以前端约束显式告知
草稿同步按钮只在当前主题为 `pixel-classic-api-safe` 时可用；其他主题下展示明确提示，不做隐式自动换主题。

原因：
- API-safe 渲染目标与手动复制目标不同，静默替换主题会造成“所见非所得”
- 用户可以清楚知道 v1 支持范围，减少故障归因成本
- 经典像素 API-safe 已经是当前最接近真实公众号草稿保真需求的版本

备选方案：
- 自动把当前主题切换为经典像素 API-safe：减少一步操作，但会让预览与最终草稿结果不一致
- 支持所有主题：实现和验收范围过大，不适合 v1

### 4. 草稿标题从首个 Markdown H1 派生，正文 HTML 去掉重复标题模块
前端以 Markdown 的首个一级标题作为微信草稿标题，并在发送前克隆 `#wechat-output` DOM，移除对应首个 H1 容器后再序列化为 `content`。

原因：
- 当前编辑器没有独立标题输入字段，这是最小变更方案
- 微信草稿标题与正文标题重复会显著影响成稿质量
- 通过克隆预览 DOM 进行裁剪，可以保持正文其余样式与当前预览一致

备选方案：
- 新增独立标题输入字段：体验更明确，但会扩大编辑器改动面
- 保留正文 H1 不裁剪：会导致标题重复，成稿体验较差

### 5. v1 使用服务端默认封面缩略图配置，而不是实现封面上传
服务端通过环境变量 `WECHAT_DEFAULT_THUMB_MEDIA_ID` 提供一个默认 `thumb_media_id`，并在草稿创建时统一带入。

原因：
- 微信草稿接口在真实场景下需要可用的封面缩略图素材 ID；v1 又明确不做封面上传 UI
- 使用预配置的默认封面能保持 v1 的“一键同步”目标，并避免阻塞正文能力上线

备选方案：
- 实现封面上传与素材管理：更完整，但超出 v1 范围
- 不带封面字段直接调用：容易因微信接口要求失败，不稳定

### 6. 后端接口最小化为一个草稿创建入口，内部自行管理 access token 缓存
新增 `POST /api/wechat/drafts`，请求体至少包含 `title` 与 `contentHtml`。服务端内部处理 access token 获取、过期刷新与微信草稿创建，不向前端暴露 token。

原因：
- 前端只需要一个“创建草稿”动作，接口越少越稳
- access token 生命周期与错误重试逻辑应全部封装在服务端

备选方案：
- 前端先调 `/token` 再调 `/drafts`：增加调用复杂度，且无安全收益

## Risks / Trade-offs

- [微信草稿接口需要有效封面缩略图] → 使用 `WECHAT_DEFAULT_THUMB_MEDIA_ID` 作为 v1 必填配置，并在缺失时直接报错阻止同步
- [仅支持经典像素 API-safe 主题会限制使用面] → 在 UI 中明确标注 v1 支持范围，保留手动复制作为通用路径
- [DOM 克隆裁剪依赖当前渲染结构] → 只移除首个 H1 容器，并为该行为添加手工回归测试
- [微信接口错误信息较难理解] → 服务端统一转换为更可操作的前端错误文案

## Migration Plan

1. 在 `server/` 增加微信公众号环境变量读取、access token 缓存与草稿创建路由
2. 在前端 `Preview` 中增加草稿同步按钮、同步状态与错误提示
3. 仅当当前主题为经典像素 API-safe 时启用草稿同步
4. 前端从 Markdown 中提取 H1 作为标题，并从同步正文中移除重复标题模块
5. 联调公众号测试号或正式号，验证成功创建草稿与错误路径
6. 若上线后出现严重同步错误，回退方式为隐藏草稿同步入口，同时保留现有复制功能
